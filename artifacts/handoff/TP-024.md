- **Status**: draft

## Objective

Claude(Odin)가 **직접 파일을 읽고 쓰고, 디렉토리를 탐색하고, 코드를 검색**할 수 있게 한다.
현재 LLM Tool은 모두 기존 Skill을 호출하는 간접 방식(get_status → status skill)인데,
이를 확장하여 **파일시스템 직접 조작 Tool**을 추가한다.

Clean Architecture 원칙에 따라:
- `core/ports/IToolExecutor.ts` — Tool 실행 인터페이스
- `core/ports/IFileSystem.ts` — 파일시스템 추상화 (테스트 가능)
- `adapters/tools/` — 실제 구현체

### 3. adapters/tools/FileSystemToolExecutor.ts (신규, ~120줄)

`IToolExecutor` 구현체. 다음 Tool을 처리:

| Tool | Input | Output |
|------|-------|--------|
| `read_file` | `{ path: string }` | 파일 내용 (최대 10,000자) |
| `write_file` | `{ path: string, content: string }` | 성공/실패 메시지 |
| `list_directory` | `{ path: string }` | 파일/디렉토리 목록 |
| `search_codebase` | `{ pattern: string, path?: string, glob?: string }` | 매칭 결과 (최대 50건) |

**경로 검증 규칙**:
- 모든 경로는 `projectRoot` 하위여야 함
- `..` 경로 순회 차단
- 심볼릭 링크 해석 후 재검증
- `.env`, `node_modules/`, `.git/` 접근 차단

```typescript
export class FileSystemToolExecutor implements IToolExecutor {
  constructor(
    private readonly fileSystem: IFileSystem,
    private readonly projectRoot: string,
  ) {}

  canHandle(toolName: string): boolean {
    return ["read_file", "write_file", "list_directory", "search_codebase"].includes(toolName);
  }

  async execute(toolCall: LLMToolCall): Promise<ToolResult> {
    const resolvedPath = this.validatePath(toolCall.input.path);
    // ... tool별 분기
  }

  private validatePath(inputPath: string): string {
    const resolved = path.resolve(this.projectRoot, inputPath);
    if (!resolved.startsWith(this.projectRoot)) {
      throw new Error(`접근 거부: 프로젝트 루트 밖의 경로 — ${inputPath}`);
    }
    // .env, node_modules, .git 차단
    const BLOCKED = [".env", "node_modules", ".git"];
    const relative = path.relative(this.projectRoot, resolved);
    if (BLOCKED.some(b => relative === b || relative.startsWith(b + path.sep))) {
      throw new Error(`접근 거부: 보호된 경로 — ${inputPath}`);
    }
    return resolved;
  }
}
```

### 5. adapters/filesystem/NodeFileSystem.ts (신규, ~80줄)

IFileSystem의 Node.js 구현체:
- `readFile` → `fs.readFile(path, "utf-8")`
- `writeFile` → `fs.writeFile(path, content, "utf-8")`
- `listDirectory` → `fs.readdir(path, { withFileTypes: true })` + `fs.stat`
- `exists` → `fs.access(path)`
- `searchContent` → `child_process.execFile("grep", ["-rn", pattern, rootPath])` 또는 순수 fs 기반 검색

### 6. ProcessCommandUseCase.ts 수정 (~50줄 변경)

```typescript
// Before: 하드코딩된 handleToolCall switch
// After: IToolExecutor 배열을 순회

constructor(
  // ... 기존 의존성
  private readonly toolExecutors: IToolExecutor[],  // 추가
) {}

private async handleToolCall(toolCall: LLMToolCall, messages: OdinMessage[]): Promise<void> {
  // approval 필요한 tool은 기존 방식 유지
  if (this.isApprovalRequired(toolCall.name)) {
    return this.handleApprovalToolCall(toolCall, messages);
  }

  // IToolExecutor 체인에서 처리 가능한 executor 탐색
  const executor = this.toolExecutors.find(e => e.canHandle(toolCall.name));
  if (executor) {
    const result = await executor.execute(toolCall, this.projectRoot);
    messages.push(this.messageRepository.addMessage({
      role: "odin",
      type: "response",
      content: result.success ? result.output : `실행 실패: ${result.error}`,
      metadata: { tool: toolCall.name },
    }));
    return;
  }

  // fallback
  messages.push(this.messageRepository.addMessage({
    role: "odin", type: "response",
    content: `지원하지 않는 도구: ${toolCall.name}`,
  }));
}
```

## Acceptance Criteria

1. **AC-1**: `read_file` tool — 프로젝트 내 파일을 읽어 내용을 반환한다
2. **AC-2**: `write_file` tool — 프로젝트 내 파일을 생성/수정한다
3. **AC-3**: `list_directory` tool — 디렉토리 내용(이름, 타입, 크기)을 반환한다
4. **AC-4**: `search_codebase` tool — 패턴 검색 결과(파일명, 줄번호, 내용)를 반환한다
5. **AC-5**: `review_saga` tool — RP 파일을 읽어 AC 항목을 파싱하여 반환한다
6. **AC-6**: 경로 안전장치 — 프로젝트 루트 밖, `.env`, `node_modules/`, `.git/` 접근 시 에러 반환
7. **AC-7**: IToolExecutor 인터페이스 — canHandle()/execute() 체인 패턴으로 기존 handleToolCall 리팩토링
8. **AC-8**: `tsc --noEmit` 0 에러
9. **AC-9**: `vitest run` 기존 테스트 + 신규 tool-executor 테스트 모두 통과
10. **AC-10**: curl /api/health 정상 응답 (서버 기동 확인)
