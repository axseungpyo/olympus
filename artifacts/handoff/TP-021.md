## Objective

비즈니스 로직을 Use Case 클래스로 캡슐화하고, 기존 routes를 얇은 Controller로 전환하여 Routes → Controller → UseCase → Repository 흐름을 완성한다.

## Scope In

### 1. core/use-cases/ — 애플리케이션 비즈니스 규칙

각 Use Case는 단일 책임. execute() 메서드 하나만 노출.
Port 인터페이스만 의존 (adapters/ 직접 import 금지).

#### agent/
- `StartAgentUseCase.ts`
  - 입력: agentName, tp, mode?
  - 동작: TP 파일 검증, 에이전트 잠금 확인, 모드 결정, 프로세스 시작
  - 의존: IAgentRepository, IProcessGateway, ITaskRepository
- `StopAgentUseCase.ts`
  - 입력: agentName
  - 동작: 프로세스 종료, PID 파일 정리
  - 의존: IAgentRepository, IProcessGateway
- `GetAgentStatusUseCase.ts`
  - 입력: (없음)
  - 동작: 모든 에이전트 상태 조회
  - 의존: IAgentRepository, ITaskRepository

#### task/
- `CreateTaskUseCase.ts`
  - 입력: CreateTaskInput
  - 동작: TP 파일 생성, INDEX.md 업데이트
  - 의존: ITaskRepository
- `ListTasksUseCase.ts`
  - 의존: ITaskRepository
- `GetTaskUseCase.ts`
  - 의존: ITaskRepository
- `UpdateTaskStatusUseCase.ts`
  - 의존: ITaskRepository
- `DeleteTaskUseCase.ts`
  - 의존: ITaskRepository

#### odin/
- `ProcessCommandUseCase.ts`
  - 입력: content (사용자 메시지)
  - 동작: 메시지 기록 → 스킬 매칭 → 실행 또는 승인 요청
  - 의존: IMessageRepository, ISkillRegistry
- `ProcessApprovalUseCase.ts`
  - 입력: approvalId, approved
  - 동작: 대기 중인 승인 처리 → 스킬 실행
  - 의존: IMessageRepository, ISkillRegistry

### 2. adapters/controllers/ — 얇은 HTTP 컨트롤러

각 Controller는 Express Request/Response를 받아서 Use Case를 호출하고 결과를 JSON으로 반환.
핸들러 함수 본문은 **10줄 이하** 목표.

- `AgentController.ts` — 기존 agent.routes.ts의 핸들러 로직 추출
- `TaskController.ts` — 기존 task.routes.ts의 핸들러 로직 추출
- `OdinController.ts` — 기존 odin.routes.ts의 핸들러 로직 추출
- `HealthController.ts` — 기존 health.routes.ts의 핸들러 로직 추출
- `McpController.ts` — 기존 mcp.routes.ts의 핸들러 로직 추출 (MCP는 UseCase 없이 Controller에서 직접 처리 가능)
- `DocumentController.ts` — 기존 document.routes.ts의 핸들러 로직 추출

### 3. routes/ 리팩토링

routes/*.routes.ts는 **라우트 정의만** 담당:
```typescript
// 예시: agent.routes.ts
export function createAgentRouter(controller: AgentController): Router {
  const router = Router();
  router.get("/api/agent/:name/health", (req, res) => controller.getHealth(req, res));
  router.post("/api/agent/:name/start", (req, res) => controller.start(req, res));
  router.post("/api/agent/:name/stop", (req, res) => controller.stop(req, res));
  return router;
}
```

### 4. di/container.ts 확장

Container에 Use Case 인스턴스 추가:
```typescript
interface Container {
  // repositories (기존)
  taskRepository: ITaskRepository;
  agentRepository: IAgentRepository;
  // ...
  // use cases (신규)
  startAgentUseCase: StartAgentUseCase;
  stopAgentUseCase: StopAgentUseCase;
  // ...
  // controllers (신규)
  agentController: AgentController;
  taskController: TaskController;
  // ...
}
```

### 5. server/index.ts 업데이트
- Container에서 controller를 꺼내서 routes에 전달

### 6. domain/ 레거시 정리
- domain/agents/agent-control.ts → StartAgentUseCase + StopAgentUseCase로 대체
- domain/odin/odin-channel.ts → ProcessCommandUseCase + ProcessApprovalUseCase로 대체
- domain/tasks/task-manager.ts → Task Use Cases로 대체
- 대체 완료 후 domain/ 파일 내용을 Use Case + Controller 호출로 교체 (또는 re-export)

## Implementation Notes

### 흐름 (After)
```
HTTP Request
  → routes/*.routes.ts (라우트 정의)
    → adapters/controllers/*.ts (입력 파싱, 응답 포맷)
      → core/use-cases/*.ts (비즈니스 로직 실행)
        → core/ports/ (인터페이스)
          → adapters/repositories/ (실제 구현)
```

### 마이그레이션 순서
1. Use Case 클래스를 먼저 생성 (기존 domain/ 코드에서 로직 추출)
2. Controller 클래스 생성 (기존 routes의 핸들러 로직 이동)
3. routes를 Controller 호출로 전환
4. Container에 Use Case + Controller 등록
5. domain/ 파일을 경량화 (Use Case를 직접 사용하거나 re-export)

### 주의사항
- OdinChannel의 pendingApprovals Map은 ProcessCommandUseCase / ProcessApprovalUseCase 사이에서 공유 필요 → 상태 관리를 IMessageRepository에 통합하거나 별도 ApprovalStore 생성
- WebSocket 핸들러도 Container/UseCase를 사용하도록 수정
- domain/odin/odin-channel.ts의 OdinChannel 클래스가 TP-020에서 이미 Container 기반으로 변경됨 → Use Case로 분해 시 이 클래스를 분해
