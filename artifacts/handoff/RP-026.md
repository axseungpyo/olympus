## Known Issues / Follow-ups
- 현재 `resumePlan()`은 스펙대로 `executePlan()` 재호출만 수행한다. approval step 자체를 승인 완료로 전환하는 저장/승인 연계는 이번 TP 범위 밖이라 포함하지 않았다.
- Plan 저장소는 메모리 기반이므로 서버 재시작 시 active plan이 유실된다.
- 런타임 `/api/health` curl 검증은 현재 Codex sandbox의 포트 바인딩 제한 때문에 직접 수행하지 못했다.

## Recommended Next Actions
1. approval UI/승인 store와 `resumePlan()`을 연결해 paused plan을 실제로 재개할 수 있게 만든다.
2. InMemoryPlanRepository를 파일 기반 또는 DB 기반 저장소로 교체해 재시작 복구를 지원한다.

## Context Digest (for Odin)
- Architecture: Planner 관련 엔티티/포트/유스케이스/어댑터를 서버 계층에 추가하고 기존 `IToolExecutor` 체인에 연결했다.
- Key entry points: `ProcessCommandUseCase`의 `create_plan`, `get_plan_status`, `PlannerUseCase.executePlan()`, `websocket/index.ts`
- Config/env: 추가 환경변수는 없다.
- Gotchas: Planner는 global `retryCount`를 사용하며, paused plan 재개는 아직 approval state transition이 없다.
