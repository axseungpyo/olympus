## Objective
서버 코드가 dashboard/lib/types.ts를 직접 import하는 역방향 의존(8개 파일)을 제거하고, src/yggdrasil/shared/types.ts를 생성하여 도메인 타입의 유일한 원천(single source of truth)으로 사용한다.

## Scope In
- `src/yggdrasil/shared/types.ts` 생성: AgentName, AgentStatus, AgentState, TaskStatus, Task, LogEntry, OdinMessage, OdinAction 등 도메인 타입 정의
- 서버 8개 파일의 import 경로를 `../dashboard/lib/types` → `../shared/types`로 변경:
  - server/agents.ts
  - server/routes.ts
  - server/control.ts
  - server/task-manager.ts
  - server/parser.ts
  - server/dependency.ts
  - server/watcher.ts
  - server/odin-channel.ts (자체 타입 정의를 shared로 이동)
- `dashboard/lib/types.ts`를 `shared/types.ts`의 re-export로 변경 (기존 dashboard import 호환)
- `dashboard/lib/constants.ts`의 AgentName import도 shared/types로 변경

## Scope Out
- 라우트 분해 (TP-017에서 진행)
- 서버 디렉토리 구조 변경 (TP-018에서 진행)
- 비즈니스 로직 변경 없음
- API/UI 동작 변경 없음

## Implementation Notes
1. `shared/types.ts` 생성 시, `dashboard/lib/types.ts`의 내용을 그대로 복사하되, UI 전용 응답 타입(StatusResponse, ChronicleResponse 등)은 dashboard에 남겨둔다.
2. `dashboard/lib/types.ts`는 `export * from "../../shared/types"` + UI 전용 타입으로 변경.
3. `server/odin-channel.ts`에 정의된 `OdinMessage`, `OdinAction` 타입이 `dashboard/lib/types.ts`에도 중복 정의되어 있으므로, shared/types.ts로 통합 후 양쪽 중복 제거.
4. `dashboard/lib/constants.ts`의 `import type { AgentName } from "./types"` 경로는 유지해도 됨 (types.ts가 re-export하므로).
5. 서버 코드의 `import { AGENT_CONFIG, AGENT_NAMES } from "../dashboard/lib/constants"` → 이 import도 해소해야 함. constants에서 서버가 필요한 항목(AGENT_NAMES, AGENT_CONFIG)을 `shared/constants.ts`로 분리하거나, server/에 별도 정의.
